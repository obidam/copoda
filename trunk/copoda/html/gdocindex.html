Introduction

COPODA stands for COllaborative Package for Ocean Data Analysis . It is a Matlab package based on the use of Matlab classes to manage, manipulate and work with hydrographic (such as temperature, salinity and biogeochemical tracers like oxygen, nitrate, phosphorus etc ...)  datas from various origins.

Available hydrographic datas are distributed in many different formats (such as netcdf, grib, mat, hsv, etc ...) and sometimes using different conventions.
Although they are very large databases aiming at uniforming such datas (hydrobase for example), scientists often have to compare, merge and work with informations coming from different sources and therefore have to develop custom routines to perform standard diagnostics.
The primary goal of COPODA is to provide an uniform and collaborative platform for diagnostics development under Matlab . Therefore it provides an interface between different datas (and as many sources) and diagnostics users would like to perform on them so that diagnostics could be written independently of the data format such as illustrated in the following schematic. 






COPODA is a set of Matlab scripts, functions and classes. 
A Matlab class is an object with specific behaviors. For example '2' is a double class instance (object) and when you type 2+2 on the prompt, Matlab calls the function plus.m (a method) to perform an addition between two double objects and return the result as another double object. Matlab allows the construction/definition of new classes and the re-definition/creation of methods. As illustrated on the schematic here above, COPODA is organized around the use of two Matlab classes: 
  • the class Transect which is going to contain informations about one campaign (or a specific set of profiles such as coming from a single measurement platform like an Argo profiler). A Transect object contains informations about the geo-localization of the datas, the platform, the cruise and the datas. Although this mix of meta and raw informations looks similar to a netcdf file, its use is more powerful as it will be illustrated in the following sections. 
  • the class Database which provides the possibility to organize, manipulate and visualize a set of Transect objects.

 

Example:
You have datas from OVIDE campaigns 2002, 2004 and 2006 as netcdf files interpolated on a common vertical grid with 1 meter resolution. You would be able to perform similar diagnostics with these campaigns because they all have the same format. Now let's imagine the following 2 situations:
1- you'd like to compare oxygen datas from OVIDE to datas from the CARINA database which is a text file with bottle samples values on an irregular grid and not available in Hydrobase.
2- you'd like to run diagnostics you developed a few years ago when you were working on datas from another campaign but those diags were for a different data format (no netcdf or no vertical interpolation), moreover you were working with oxygen datas in ml/l and now you use mumol/kg.
3- you'd like to look at Argo profiles close to the OVIDE cruise track.


In the first situation you would have to interpolate and convert CARINA datas to match OVIDE format. In the second situation you would have to re-write your diagnostics and manage the unit conversion. In the the third situation you would have to select specific Argo profiles, manage different grid and file formats. By using COPODA these painful steps would be perform under the hood and automatically in a few seconds, allowing you to focus on diagnostics improvements and physics finally improving your productivity.


Matlab classes: a brief overview

In the MATLAB language, each value is assigned to a class. For example, creating a variable with an assignment statement (the operator =) constructs a variable of the appropriate class:

>> a = 7;
>> b = 'some string';
>> whos
  Name      Size            Bytes  Class     

  a         1x1                 8  double
  b         1x11               22  char
Basic commands like whos display the class of each value in the workspace. This information helps MATLAB users to recognize that some values are characters and display as text while other values might be double, single, or other types of numbers. Some variables can contain different classes of values like cells.
You can view classes as new data types having specific behaviors defined for the class. For example, a polynomial class might redefine the addition operator (+) so that it correctly performs the operation of addition on polynomials. In our case the addition operator between two Transect objects could create a Database object. Operations defined to work with objects of a particular class are known as methods of that class. When you define a class, you can overload special MATLAB functions (plus.m for the addition operator) that are called by the MATLAB runtime when those operations are applied to an object of your class.


MATLAB classes use the following words to describe different parts of a class definition and related concepts.

  • Class definition — Description of what is common to every instance of a class.
  • Properties — Data storage for class instances
  • Methods — Special functions that implement operations that are usually performed only on instances of the class
  • Objects — Instances of classes, which contain actual data values stored in the objects' properties
  • Packages — Folders that define a scope for class and function naming


These are general descriptions of these components and concepts. The online documentation here describes all of these components in detail.

A class instance is created by two different ways of calling the class constructor. 
A call with no argument will create the object with default properties:
>> T = transect;
A call with pairs of properties/properties-values will fill specified fields:
>> T = transect('creator','John Doe');


The Transect class

One instance of a Transect class is called a Transect object. A Transect gather information and measurements from a single platform, i.e. a cruise or a unique Argo profiler.
All Transect objects have the following properties:

Basic properties

Basic properties are source, creator, file, file_date, created  and modified. They all are self explanatory and can be accessed/modified by simple classic structure mean like:
>> T.source
>> T.source{:}
>> T.source{1:end}
>> T.modified = now
>> T.creator = 'Your Name'
>> T.creator(1:4) = 'Mine'
Note that you can use either {} or () to access the content for these properties.

The 'cruise_info' property 

This property is itself a class similarly called cruise_info. I acknowledge that the use of similar names for a Transect object property and its inner content as another class may be misleading. To avoid any confusion between the Transect property and its value (both having the same name here), we'll color in blue the reference to the Transect property and in red the reference to its value as a (sub)-class. The cruise_info class has the following properties:

Properties Type Description
NAME string Name of the cruise
PI_NAME string Name of the Principal Investigator (PI) of the cruise
PI_ORGANISM string PI's affiliation
SHIP_NAME string Name of the ship
SHIP_WMO_ID string WMO identifier of the ship. They can be found here: http://www.wmo.int/pages/prog/amp/mmop/wmo-number-rules.html
DATE datenum (2x1 array) First and last days of the cruise, as return by datenum
N_STATION double Number of stations/profils performed during the cruise

The use of another class instead of a classic structure to gather these informations in a Transect object was motivated by the eventual possibility of adding exotic additional informations or manipulating string contents (although not exploited at this time). To the end user, this doesn't make any difference.

The 'geo' property

This property will gather all informations about axes/geo-localization of datas. The default list of fields to provide is:
Fields Type Dimensions Description
STATION_NUMBER double N_STATIONS x 1 Individual station number IDs
STATION_DATE datenum N_STATIONS x 1 Date of the station, we assume here it doesn't take more than a day to perform a station.
LATITUDE double N_STATIONS x 1 Latitude (-90 to 90) of stations
LONGITUDE double N_STATIONS x 1 Longitude of stations (no convention but 0:360 is appreciated)
POSITIONING_SYSTEM string variable
 The positioning system used, assumed to be similar for all the stations
PRES double N_STATIONS x N_LEVELS or N_STATIONS x 1 The pressure (dbar) matrix of samples
MAX_PRESSURE double N_STATIONS x 1 The maximum pressure at each stations (for each profiles)
DEPH
N_STATIONS x N_LEVELS or N_STATIONS x 1 The depth (meter) matrix of samples


To define such fields you can use a simple syntax like:

>> T = transect;
>> T.geo.STATION_NUMBER = 1:123;
This list is customizable. For example you can add the centered area between stations and vertical grids (useful for a tracer transport computation) or anything else because there are no restrictions on the content of the geo property.

Note that is some cases the vertical grid is the same for all stations, therefore PRES and DEPH could be of N_STATIONS x 1 dimension only. In the case of an irregular vertical grid, N_LEVELS will be the maximum number of levels possible and for stations with less levels, values will be continuously filled starting from the first index, the remainder being filled with NaNs.
With this version of COPODA, all datas are supposed to be localized on the same grid !

 These fields are OData class instances (see hereafter) containing the datas.


STATION_PARAMETERS

The property STATION_PARAMETERS is read only and is automatically filled. When a call to this property is made, a scan of the Transect object data property is performed and the list of OData objects with property name or long_name defined is returned. If none of the OData objects has a name defined, a NaN is returned. STATION_PARAMETERS is basically the default output of the Transect method datanames, ie if T is a Transect object a call to T.data.STATION_PARAMETERS gives the same output as datanames(T).

PARAMETERS_STATUS

The field PARAMETERS_STATUS contains the status of datas. This status can be:

  • R for Real. The corresponding OData object really contains the values of the fields. This is the default value. Real, doesn't stand for Real/Imaginary numbers !
  • V for Virtual. The corresponding OData object has all properties defined (name, unit, etc...) but the values are not stored in memory. In this case, values are dynamically computed when needed. For example, the AOU is the difference between the oxygen solubility and concentration, so this difference is computed every time a call to the AOU content is made. Obviously, the more often you have access to a variable, the less efficient is the use of such virtual variables. This status is only suitable for variables computed using a formulae, temperature TEMP cannot be a virtual variable for example.


By default the status is set to Real when adding a variable without specifying its status.

To read the status of a variable use the function dstatus:

>> dstatus(T,'TEMP')
ans =
R

and to get the index of TEMP among STATION_PARAMETERS to set its status, use the input option 1:

>> dstatus(T,'TEMP',1)
ans =
    24
>> T.data.PARAMETERS_STATUS(dstatus(T,'TEMP',1)) = 'R';

Note that the status is defined only in capital letters. Type help transect/dstatus for more informations.

Working with Transect objects: methods

Methods associated with a Transect object technically refer to all the functions listed into the directory:
/copoda/@transect/

However, we will refer to them according to the three different types:
  • Modified Matlab methods: these are Matlab built-in functions which are specifically called when associated to a Transect object. For example, the function size(T) should return 1x1 if not defined for a Transect object, but it's actually defined and return the number of stations x number of vertical levels.
  • Specific methods: these are functions built specifically to ease information's manipulation or retrieval from a Transect object. For example, isdata(T,'TEMP') will indicate if the temperature is an available variable in the Transect object T.
  • Specific diagnostics: these are also functions built specifically for a Transect object2 but they refer to functions providing a high level of information manipulations. For example the method omp will perform  an Optimum Multiparameter analysis on the Transect object datas.

To present an exhaustive list of the methods available to manipulate a Transect object, we'll assume the user has downloaded the netcdf sample file located here:
http://w3.ifremer.fr/lpo/base_hydro/mlt/AR21_dep.nc 
which contains datas of the WOCE 1998 AR21 section across the Eastern North-Atlantic.

How to add a test for validation ?

All tests are functions located here:

/copoda/@transect/private/

and are Matlab files beginning by: test_
The validate method for Transect object performs a scan of the private directory, looking for test_*.m files when we call: validate(transect,'list').

To create your own test, you can start using the test function template called:

 /copoda/@transect/private/template_test.m
which highlight the test file structure:

The general behavior of a test function is as follow:
  • without input arguments, the test must returns:
    • output 1: the test ID (a unique number assigned to each test once and for all).
    • output 2: the test description (a cell of string)
  • with input arguments:
    • input 1: the Transect object to validate
    • input 2: 0 or 1 if we want informations to be displayed on screen
    • input 3: 0 or 1 if we want to try to fix the Transect
    • output 1: the boolean result of the test
    • output 2: a message structure
    • output 3: the boolean result of the fixing process
    • output 4: the eventually fixed Transect object

end %functiontemplate_test

Standard workflow

Converting datas to Transect and/or Database objects

The first step, and may be the more painful one, is to write the script which is going to convert a data set to a Transect and/or Database object in Matlab. To do so a perfect knowledge of theses classes is necessary and the reader is referred to section X for their complete descriptions. 
The good news is that when such a conversion script is written, it's done once and for all. The bad news is that there should be as many scripts as input data formats.
As of now, the package comes with conversion scripts to manage:
- Hydrobase netcdf files
- Hydrolpo netcdf files
- CARINA text files
- Argo netcdf profiles
These already provide a lot of profils ...
A Database object is a collection of Transect objects, therefore this is a very flexible concept because one would like to create a database with profils having oxygen datas and another to create a database with profils within a particular geographic area.

Develop diagnostics and methods for Transect and/or Database objects

The package comes with pre-defined simple diagnostics but the goal here is that you and your coworkers develop your own. Once you'll know the structure of Transect and Database objects, you'll know how to write diagnostics and specific methods to manage them.
